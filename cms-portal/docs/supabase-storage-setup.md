# Supabase Storage Setup

This portal is now prepared to store files in Supabase Storage using a structured path format.

## Bucket

Create one private bucket:

- Name: `cms-files`

If you want a different bucket name, set:

- `NEXT_PUBLIC_SUPABASE_STORAGE_BUCKET=your-bucket-name`

## Folder Structure

Files are stored like this:

```text
users/{username}/tasks/{taskId}/attachments/{year}/{month}/{uuid}-{fileName}
```

Examples:

```text
users/admin/tasks/123e4567-e89b-12d3-a456-426614174000/attachments/2026/03/uuid-brief.pdf
users/ayesha/tasks/123e4567-e89b-12d3-a456-426614174000/attachments/2026/03/uuid-design.png
```

This keeps every user's data grouped cleanly.

Account files are stored like this:

```text
users/{username}/accounts/{accountId}/files/{year}/{month}/{uuid}-{fileName}
```

User avatars are stored like this:

```text
users/{username}/profile/avatar/{year}/{month}/{uuid}-{fileName}
```

## Why Signed Upload URLs

This portal does **not** use Supabase Auth for end users.
Because of that, browser uploads should use signed upload URLs generated by the server.

That is now implemented for task attachments.

## What You Need To Do In Supabase

### 1. Create the bucket

In Supabase Dashboard:

1. Open `Storage`
2. Click `New bucket`
3. Name it `cms-files`
4. Keep it `Private`

### 2. No open public upload policy needed

Because uploads now use signed upload URLs from the server, you do **not** need to allow anonymous browser uploads.

### 3. Service role key

Make sure this env var exists in your app:

```env
SUPABASE_SERVICE_ROLE_KEY=...
```

This is required because the server creates signed upload URLs and signed download URLs.

### 4. Optional bucket env

If you use the default bucket name `cms-files`, you do not need this.
If not, add:

```env
NEXT_PUBLIC_SUPABASE_STORAGE_BUCKET=cms-files
```

## Extra DB Table For Account Files

Run this SQL in Supabase SQL editor:

```sql
-- file: docs/supabase-account-files.sql
create table if not exists public.account_files (
  id uuid primary key default gen_random_uuid(),
  account_id text not null references public.accounts(customer_id) on delete cascade,
  file_name text not null,
  file_size bigint null,
  mime_type text null,
  storage_path text not null,
  uploaded_by text not null references public.users(username) on delete restrict,
  created_at timestamptz not null default now()
);

create index if not exists account_files_account_id_idx
  on public.account_files (account_id, created_at desc);
```

## Next Recommended Migration

If you want to fully replace old Drive usage, next good steps are:

1. Migrate any old base64 avatars into Supabase Storage if you want to clean old records
2. Migrate any old external Drive links into `account_files`
3. Remove old `allowed_drive_folders` / `drive_access_level` fields from DB and auth types after you no longer need backward compatibility

## Notes

- Existing task attachments stored in the old bucket/path will stay there until you migrate them.
- New uploads will use the new structured storage path.
